How I Built a RAG Chatbot That Doesn't Cost a Rupee to Run

The full story behind ChapterChat — why I swapped OpenAI for a locally-run LLM, how I structured the retrieval pipeline, and the three mistakes I made along the way that you don't have to.

When I started building ChapterChat, my plan was simple: upload a PDF, ask questions about it, get smart answers. Classic RAG. What I didn't plan for was how quickly OpenAI API costs add up when you're a student iterating on a project twenty times a day.

After blowing through my free credits in under a week, I made a decision that turned out to be one of the best architectural choices of the project: run the LLM locally using Ollama. Zero cost per query, no rate limits, and a bonus — I actually learned more about how the inference stack works.

What is RAG, really?

RAG stands for Retrieval-Augmented Generation. The idea is that instead of asking an LLM to answer from memory (which leads to hallucinations), you first retrieve the relevant pieces of text from your own documents, then hand those pieces to the LLM as context for its answer.

Three steps: chunk your document → embed the chunks → retrieve the most relevant ones at query time → generate an answer using those chunks as context.

The Architecture

Step 1 — Document Ingestion

The first challenge is turning a PDF into something a model can search through. I used LangChain's document loaders to parse the raw text, then split it into chunks using RecursiveCharacterTextSplitter:

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ".", " "]
)
chunks = splitter.split_documents(docs)

The chunk_overlap=50 is important — it prevents answers from being cut off at chunk boundaries. I learned this the hard way when my chatbot kept giving half-answers about anything that spanned a page break.

Step 2 — Embeddings & Vector Store

Each chunk gets converted into a vector (a list of numbers that captures its meaning) and stored in ChromaDB. I used Hugging Face's all-MiniLM-L6-v2 model for embeddings — fast, accurate, and completely free to run locally:

from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma

embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2"
)
vectorstore = Chroma.from_documents(
    chunks, embeddings, persist_directory="./chroma_db"
)

Step 3 — Swapping OpenAI for Ollama

This was the key move. Ollama lets you run open-source LLMs like llama3 directly on your machine. LangChain has a native integration:

from langchain.llms import Ollama

llm = Ollama(model="llama3")  # pull with: ollama pull llama3

💡 Tip: If you're on a machine with less than 8GB of RAM, use llama3:8b-instruct-q4_0 — the quantised version runs surprisingly well even on modest hardware.

Step 4 — The Retrieval Chain

Wiring it together with LangChain's RetrievalQA:

from langchain.chains import RetrievalQA

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever(
        search_kwargs={"k": 4}  # top 4 chunks
    ),
    return_source_documents=True
)

result = qa_chain({"query": "What are the main themes?"})
print(result["result"])

Three Mistakes I Made (So You Don't Have To)

  • Chunks too large. I started with chunk_size=2000. The LLM kept getting confused by too much context in a single chunk. 400–600 tokens is the sweet spot for most documents.
  • No chunk overlap. My first version split chunks perfectly cleanly with no overlap. Answers about topics that crossed a chunk boundary were always incomplete. Add at least 10% overlap.
  • Forgetting to persist ChromaDB. Every time I restarted the server, I was re-embedding the entire document. Add persist_directory from day one.

What I'd Add Next

If I were building ChapterChat v2, I'd add hybrid search — combining semantic (vector) search with keyword (BM25) search. Semantic search is great for meaning, but it misses exact terms like model names or version numbers. Hybrid covers both.

I'd also look at re-ranking the retrieved chunks before passing them to the LLM — a second smaller model that scores chunks for relevance before they reach the generator. It's an extra step but measurably improves response quality.
