RAG, Explained Like You're a Developer Who's Never Used It

Retrieval-Augmented Generation sounds fancy. It's not. Here's a plain-English breakdown of how it works, why it exists, and real code examples from my own projects — no ML PhD required.

If you've spent any time in AI circles lately, you've heard "RAG" thrown around constantly. It's one of those terms that sounds intimidating but is actually a pretty intuitive idea once you strip the jargon away.

I'll explain it the way I wish someone had explained it to me before I built ChapterChat.

The Problem RAG Solves

Large language models — ChatGPT, Claude, llama3 — are trained on enormous amounts of text from the internet. They're impressive at reasoning, writing, and summarizing. But they have a fundamental limitation: they only know what they were trained on.

Ask ChatGPT about a document you just uploaded, and it'll do its best — but it's actually guessing based on patterns, not reading your document. Ask it about an event from last week, and it might hallucinate an answer confidently. The model's knowledge is frozen at its training cutoff.

RAG is the solution. Instead of asking the LLM to answer from memory, you retrieve the relevant information first, then hand it to the LLM as context. The LLM isn't guessing anymore — it's reading.

Breaking Down Each Step

1 — Chunking

You can't feed an entire 200-page PDF to an LLM at once (context windows have limits, and it's slow and expensive). So you split the document into smaller pieces — "chunks" — of a few hundred words each.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,   # ~500 tokens per chunk
    chunk_overlap=50  # overlap to avoid cut-off answers
)
chunks = splitter.split_documents(docs)

The overlap is critical. If a sentence starts at the end of chunk 3 and finishes at the start of chunk 4, overlap ensures you don't lose that context.

2 — Embeddings

An embedding is a list of numbers — a vector — that represents the meaning of a piece of text. The clever part: texts with similar meanings end up with vectors that are mathematically close to each other.

So "the cat sat on the mat" and "a feline rested on a rug" would have very similar embeddings, even though the words are totally different.

from langchain.embeddings import HuggingFaceEmbeddings

# Free, runs locally, surprisingly good
embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2"
)

3 — The Vector Store

All those chunk embeddings need to live somewhere searchable. That's what a vector database does — it stores vectors and lets you find the ones closest to any query vector, extremely fast. I used ChromaDB for ChapterChat.

from langchain.vectorstores import Chroma

vectorstore = Chroma.from_documents(
    chunks,
    embeddings,
    persist_directory="./chroma_db"  # saves to disk
)

4 — Retrieval

When a user asks a question, you embed that question too — and search the vector store for the chunks whose vectors are closest to the question's vector. These are the most semantically relevant chunks.

retriever = vectorstore.as_retriever(
    search_kwargs={"k": 4}  # return the top 4 chunks
)

relevant_chunks = retriever.get_relevant_documents(
    "What are the main arguments in chapter 3?"
)

5 — Generation

Finally, you hand the retrieved chunks to the LLM as context, and ask it to answer the question. The LLM is now answering from real, specific evidence rather than its training memory.

from langchain.chains import RetrievalQA
from langchain.llms import Ollama

llm = Ollama(model="llama3")

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True
)

response = qa_chain({"query": "What are the main arguments in chapter 3?"})
print(response["result"])

Why Not Just Use Fine-Tuning Instead?

A common question. Fine-tuning means training the model on your specific data so it "knows" it. RAG means keeping a generic model and retrieving relevant context at query time. Here's when each makes sense:

  • Use RAG when your knowledge base changes frequently, or when you need citations and traceability. It's cheaper and faster to update.
  • Use fine-tuning when you need the model to adopt a specific style, tone, or domain-specific reasoning pattern — not just recall facts.
  • Use both when you have the budget and need both domain adaptation and fresh knowledge retrieval.

💡 For most student projects and early-stage products, RAG is the right starting point. It's faster to build, cheaper to run, easier to debug, and you can see exactly what information the model is using to answer.

Where to Go Next

Once you've got basic RAG working, the two most impactful improvements are:

  • Hybrid search — combine semantic (vector) retrieval with keyword (BM25) search. Great for queries that include proper nouns, version numbers, or exact terms.
  • Re-ranking — after initial retrieval, use a second model to score chunks for relevance before passing them to the generator. Measurably improves quality.
